# 公开版 AI Specification：Document-to-Project Agent

> 目标：定义 AI 能力边界、输入输出契约、确定性规则、质检职责和降级策略。本文是脱敏后的工程协作规格，不包含生产 Prompt、模型密钥或内部接口。

## 1. Agent 定位

Agent 将 Excel 中的项目事实转换为结构化项目草稿。它不是自由生成项目内容的聊天机器人，也不能绕过用户确认直接创建正式项目。

设计原则：

- **事实优先**：任务、日期、负责人来自源文件，不凭空补齐。
- **结构稳定**：全量节点由确定性代码生成。
- **语义增强**：AI 负责表头理解、分组建议、命名检查和异常解释。
- **可追溯**：每个任务保留来源工作表与行号。
- **可降级**：无法确认的字段留空或进入默认分组，并产生 warning。

## 2. 输入契约

### 2.1 支持输入

- 文件类型：`.xlsx`
- 推荐结构：单行表头、每行一个任务事实。
- 公开 Demo 表头：`模块、卡片、任务、负责人、开始日期、截止日期、备注`

### 2.2 标准化中间输入

```json
{
  "project_name": "市场活动排期",
  "rows": [
    {
      "module": "筹备",
      "card": "内容准备",
      "task": "完成活动主文案",
      "owner": "内容负责人",
      "start_date": "2026-07-01",
      "end_date": "2026-07-03",
      "notes": "首轮评审前完成",
      "source_sheet": "项目计划",
      "source_row": 2
    }
  ]
}
```

## 4. Code 与 AI 的职责边界

| 环节 | Code | AI |
| --- | --- | --- |
| 文件读取 | 读取 workbook、单元格类型和工作表 | 不参与 |
| 表头识别 | 候选列、别名匹配、位置约束 | 处理模糊表头与语义别名 |
| 节点生成 | 每行确定性创建任务，稳定分组与 ID | 不全量重建 |
| 字段校验 | schema、日期、必填项、覆盖率 | 判断语义合理性 |
| 修正 | 可逆、局部、规则明确的修正 | 提出建议并说明置信度 |
| 写入 | 幂等、权限、重试、审计 | 无权直接调用 |

关键约束：AI 不能删除源行、增加无来源任务或更改事实日期；若提出修改，必须提供目标节点、原因和原值。

## 5. 字段映射

| 目标字段 | 常见表头 | 规则 |
| --- | --- | --- |
| `module` | 模块、阶段、一级分类 | 空值进入“待分类” |
| `card` | 卡片、工作包、里程碑 | 空值进入“任务清单” |
| `task` | 任务、事项、工作内容 | 必填；空值行跳过 |
| `owner` | 负责人、执行人、Owner | 保留文本，不推断真实账号 |
| `start_date` | 开始日期、计划开始 | 输出 `YYYY-MM-DD` |
| `end_date` | 截止日期、计划完成 | 输出 `YYYY-MM-DD` |
| `notes` | 备注、说明 | 去除首尾空白 |

## 6. 生成规则

### 6.1 项目名

优先级：

1. 用户在预览页输入的名称。
2. 工作簿中的明确标题。
3. 文件名去除扩展名。

禁止使用真实客户名自动拼接宣传性描述。

### 6.2 模块与卡片

- 按源行首次出现的顺序保留。
- 不因名称相似自动合并，避免误合并。
- 缺失时使用默认分组，同时记录使用次数。

### 6.3 任务

- 一个有效源行至少映射一个任务。
- 空任务名行跳过，并输出 `MISSING_TASK_TITLE`。
- 任务 ID 按有效输入顺序稳定生成。
- `source_ref` 保存源工作表和 Excel 行号。

## 7. Warning 规则

| code | 级别 | 触发条件 | 处理 |
| --- | --- | --- | --- |
| `INVALID_ROW` | warning | 行不是可解析对象 | 跳过 |
| `MISSING_TASK_TITLE` | warning | 任务名为空 | 跳过 |
| `DEFAULT_MODULE_USED` | info | 模块为空 | 默认分组 |
| `DEFAULT_CARD_USED` | info | 卡片为空 | 默认分组 |
| `INVALID_DATE` | warning | 日期不是 ISO 格式 | 保留并要求确认 |
| `DATE_RANGE_CONFLICT` | warning | 开始晚于结束 | 要求确认 |
| `UNSUPPORTED_LAYOUT` | blocking | 找不到稳定任务区域 | 停止创建 |

warning 必须可定位到源行；禁止只返回“生成结果可能不准确”。

## 8. 置信度

置信度只描述字段或分组判断，不代表项目整体可用性：

- `high`：直接表头匹配且字段类型合法。
- `medium`：语义别名命中或需要轻量归一化。
- `low`：依赖上下文推断，需要用户确认。

发布判断不能只看平均置信度，还需要覆盖率、严重 warning、稳定性和人工复核。

## 9. 失败与降级策略

1. 解析失败：返回文件级错误，不进入 AI 阶段。
2. 表头模糊：展示候选映射，请用户确认。
3. 层级缺失：使用默认分组，不虚构业务层级。
4. 日期冲突：保留事实并标记，不自动交换日期。
5. 超复杂表格：生成部分预览并说明未覆盖区域，或阻断创建。
6. AI 超时：回退到确定性映射结果，保留待人工确认项。
7. 写入失败：保留 preview artifact 和幂等键，允许重试。

## 10. 可观测性

每个任务记录：

- `task_id`、版本、输入文件哈希。
- 各阶段耗时与最终状态。
- 工作表数、源行数、映射任务数。
- warning 分布和默认分组使用次数。
- 用户是否修改预览及修改字段。

日志不存储原始文档正文、真实姓名或账号。

## 11. 评测与验收

### 自动评测

- Schema 合规率。
- 任务覆盖率。
- 字段命中率。
- 来源可追溯率。
- 同输入多次运行的结构 diff。
- P50 / P95 生成耗时。
- 阻断级 warning 计数。

### 人工复核

- 层级是否符合业务理解。
- 卡片分组是否可执行。
- 是否存在“看起来合理但事实错误”的内容。
- 预览是否能以可接受成本完成修正。

### 验收原则

自动高分不直接等于可发布。任何数据遗漏、父子关系错误或错误写入风险都可以覆盖平均分，触发阻断。

## 12. Prompt / Workflow 约束

- Prompt 只接收必要字段和局部上下文。
- 输出必须符合版本化 JSON Schema。
- 禁止输出源文件中不存在的人员、日期和任务。
- AI 修正应是 patch，而不是重新生成整个项目。
- 超时或解析失败不得重试无限次。
- 所有最终写入仍由工程层进行 schema、权限与幂等校验。
