# 项目案例复盘：从业务文档到结构化项目

## 1. 背景

业务团队习惯用 Excel 维护项目计划、任务清单、施工排期和活动节奏。表格灵活，但当计划需要进入项目管理系统协作时，通常需要重新创建项目层级、分组和任务，迁移成本高且容易漏项。

本项目希望让用户上传文档后，快速得到一个可预览、可编辑的项目骨架，并在确认后完成创建。

## 2. 我的工作重点

本作品集重点呈现以下工作：

- 将模糊需求拆分为产品流程、AI 能力规格和评测标准。
- 定义项目、模块、卡片、任务、子任务等层级及字段映射。
- 设计 Code 与 LLM 的职责边界、异常降级和可追溯机制。
- 建立自动评测 + 人工复核体系，并通过回归数据推动迭代。
- 根据用户反馈补齐预览编辑、后台执行、结果恢复和通知链路。
- 将项目方法论、复盘和流程图沉淀为可复用文档资产。

## 3. 核心用户问题

早期验证显示，“能生成”并不等于“可使用”：

1. 同一份文档多次生成，结果可能不一致。
2. 复杂表格中可能出现漏任务、错误分组或父子关系丢失。
3. 生成时间较长时，用户不知道是否需要继续等待。
4. 用户关闭页面后任务可能中断，之前的等待被浪费。
5. 只通过对话修改大项目结构，确认成本较高。
6. 自动评分较高的结果，人工仍可能判断为不可用。

## 4. 产品方案

### 4.1 预览优先的主流程

```text
选择文件
  -> 创建后台生成任务
  -> 展示进度与可恢复入口
  -> 生成项目预览
  -> 用户编辑或确认
  -> 创建正式项目
  -> 跳转项目详情
```

关键点：

- 生成与创建分离，避免 AI 结果未经确认直接写入业务系统。
- 允许用户离开当前页面，后台继续执行。
- 生成完成后提供统一恢复入口。
- 预览页支持结构化修改，复杂调整再进入对话。
- 创建操作需要幂等保护，防止重复项目和重复任务。

### 4.2 AI 侧输出

AI 层输出统一的结构化对象：

- 项目：名称、描述、来源信息。
- 模块：大阶段或一级业务分组。
- 卡片：可执行工作包或二级分组。
- 任务/子任务：标题、负责人、时间、备注、标签等。
- `confidence`：整体或局部置信度。
- `warnings`：缺失字段、日期冲突、默认分组、歧义等问题。
- `source_ref`：用于回到源表位置的追溯信息。

### 4.3 失败与降级

- 表格信息不完整时，使用明确的默认模块/卡片，并生成 warning。
- 日期或层级存在冲突时保留事实值，不静默“猜对”，交由预览或质检处理。
- 文档不适合生成项目时，返回可理解的失败原因。
- 高风险写入动作始终在用户确认后执行。

## 5. 关键技术决策

最重要的架构调整，是从“让 LLM 阅读整张表并直接生成全部任务”改为：

```text
文档事实源
  -> Code 提取表格语义
  -> Code 确定性生成全量节点
  -> Code 校验节点覆盖与字段合法性
  -> LLM 做轻量语义质检
  -> Code 组装最终创建入参
```

原因：

- 全量重建容易因上下文长度、随机性和模型超时产生漏项。
- 确定性节点更容易复算覆盖率、定位错误和局部修复。
- LLM 更适合处理命名合理性、分组语义和异常解释，而不是承担所有结构生成。
- 最终写入前仍由代码校验 schema 和业务约束。

详细设计见 [Agent 架构设计](architecture.md)。

## 6. 评测结果与决策

一次 v0.3 回归覆盖 20 个脱敏样例：

| 指标 | 结果 |
| --- | ---: |
| AI 自动评测平均分 | 89.5 |
| 人工复核平均分 | 83.1 |
| 人工通过 | 13 |
| 有条件通过 | 2 |
| 不通过 | 5 |
| 平均耗时 | 40.1 秒 |

在与上一版本共有的 10 个样例中，人工均分从 76.6 提升到 88.0，说明迭代总体有效；但人工仍发现分组、层级关系和业务可用性问题。

因此当时的产品判断不是“平均分接近 90，可以发布”，而是：

- 保留版本改进结论。
- 将自动评测与人工判断分歧列为评测系统缺陷。
- 优先修复阻断/严重样例。
- 在性能和恢复链路达标前，不进入稳定对外发布。

## 7. 复盘

### 做对的事情

- 把 PRD、AI Spec 和评测标准拆开，减少产品、研发和模型团队的口径混淆。
- 用结构化预览承接 AI 的不确定性，让用户拥有最终控制权。
- 以真实样例回归而不是单次演示判断质量。
- 同时记录自动分、人工分、耗时和缺陷严重度。
- 用版本对比证明迭代是否真的改善。

### 暴露的问题

- 自动指标对业务语义覆盖不足，容易高估复杂样例。
- 平均分会掩盖少数严重失败，必须设置硬性阻断条件。
- “生成速度”不能只作为观察指标，它直接影响产品体验和转化。
- 复杂表格需要分层支持范围，不能用一个统一成功率描述全部能力。

### 下一步

- 增强路径级匹配和父子关系评测。
- 将稳定性从单次结果升级为同文档多次运行的 diff 指标。
- 分别统计基础任务表、标准项目计划和复杂工程计划。
- 完善后台任务状态机、恢复入口和失败重试。
- 对低置信度节点提供局部确认，而不是要求用户重新生成整个项目。
