# 匿名评测报告：为什么自动高分不等于可发布

## 1. 数据说明

本仓库提供 20 条脱敏后的逐样例结果：

- [原始 CSV](../evaluation/anonymized_results.csv)
- [复算脚本](../evaluation/summarize_results.py)
- [机器可读汇总](../evaluation/summary.json)

客户名、文件名、行业和业务内容已移除，仅保留复杂度层级、自动评分、人工评分、结论和耗时。数据用于展示评测方法与版本判断，不代表所有业务场景的普遍效果。

复算命令：

```powershell
python evaluation/summarize_results.py
```

## 2. v0.3 快照

| 指标 | 结果 |

| 样例数 | 20 |

| 人工复核平均分 | 83.1 / 100 |
| 人工结论 | 13 通过 / 2 有条件通过 / 5 不通过 |

与上一版本共有的 10 个回归样例中，人工均分从 76.6 提升到 88.0，通过数从 5 提升到 7，P0/P1 问题从 5 个降到 1 个。由于公开 CSV 不保留可关联旧版本的样例身份，这组对比只作为脱敏后的版本汇总，不在脚本中伪造逐条关联。

## 3. 关键发现

### 3.1 自动评测偏乐观

20 个样例中有 4 个出现“自动结论通过，但人工结论不通过”。典型原因不是 JSON 格式错误，而是：

- 卡片分组不符合实际协作方式。
- 父子关系看似完整，但业务语义错位。
- 任务字段齐全，却不能直接执行。
- 平均分掩盖了少数严重漏项。

因此，自动指标适合作为回归筛查，不能单独替代业务复核。

### 3.2 平均耗时掩盖长尾

平均耗时 40.1 秒，但最慢样例达到 252.5 秒。只汇报平均值会低估用户等待和任务中断风险，因此产品方案增加了后台任务、最小化和恢复入口。

### 3.3 复杂度需要分层

| 层级 | 数量 | 自动均分 | 人工均分 |
| --- | ---: | ---: | ---: |
| Tier 1 | 3 | 95.6 | 81.1 |
| Tier 2 | 13 | 88.6 | 81.6 |
| Tier 3 | 4 | 87.9 | 89.7 |

层级均分不是单调变化，说明样例量仍小，而且“复杂度”不能替代具体失败类型分析。下一轮应增加各 Tier 样例数，并按布局、字段、层级和跨表依赖建立能力矩阵。

## 4. 版本决策

该版本没有因为自动均分接近 90 就直接宣布稳定发布。决策是：

1. 保留“继续修复与灰度验证”结论。
2. 将自动/人工分歧列为评测体系缺陷。
3. 优先修复阻断和严重样例。
4. 增加稳定性、路径关系和性能长尾指标。
5. 在预览与后台恢复链路达标前，不扩大开放范围。

## 5. 下一轮评测

- 同一文档重复运行 3 次，计算节点级结构 diff。
- 增加路径匹配、父子关系和源行追溯指标。
- 分开统计基础任务表、标准项目计划和复杂工程计划。
- 报告 P50 / P95，而不是只报告平均耗时。
- 对自动高分、人工不通过样例建立 hard-case 回归集。
- 统计用户在预览中修改了哪些字段，以验证评测指标是否覆盖真实成本。
